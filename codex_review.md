# Codex PR Review

Branch reviewed: `feat/milestone-m0-m8-implementation`  
Base compared: merge-base with `origin/main` at `9de0b2625562f4c2a0ac6d2689256cecfec31de7`

## Findings

### 1. High: history-aware routing looks up a different fingerprint shape than ingest writes

The `routing_feedback` path appears effectively inert for most real runs.

The ingest side records routing feedback from finalized runs using `TaskFingerprint.from_document/1`, which includes the actual toolset used during the run:

- `apps/lemon_core/lib/lemon_core/memory_ingest.ex:135`
- `apps/lemon_core/lib/lemon_core/task_fingerprint.ex:66`
- `apps/lemon_core/lib/lemon_core/task_fingerprint.ex:101`

That produces context keys shaped like:

```text
family|tool1,tool2|workspace
```

The router lookup side, however, builds a pre-execution fingerprint with only:

- classified prompt
- `cwd` as workspace
- no toolset at all

See:

- `apps/lemon_router/lib/lemon_router/run_orchestrator.ex:563`
- `apps/lemon_router/lib/lemon_router/run_orchestrator.ex:573`

That means the router usually searches with a context key like:

```text
family|-|workspace
```

while the feedback store mostly contains keys like:

```text
family|bash,read_file|workspace|provider|model
```

`RoutingFeedbackStore.best_model_for_context/1` does prefix matching on the full context key:

- `apps/lemon_core/lib/lemon_core/routing_feedback_store.ex:470`

So unless a prior run used no tools, the router’s lookup key will not match the stored feedback rows. The feature can therefore collect data without meaningfully affecting selection.

Why tests miss this:

- `best_model_for_context/1` tests use synthetic keys with `-` toolsets instead of keys generated by actual ingest behavior:
  - `apps/lemon_core/test/lemon_core/routing_feedback_store_test.exs:229`
- there is no end-to-end test from finalized run -> stored fingerprint -> router lookup.

### 2. High: `search_memory` can silently broaden scoped searches to the entire memory store

The new `search_memory` tool advertises scoped search, but for `agent` and `workspace` it can fall back to global search.

The tool schema exposes:

- `query`
- `scope`
- `limit`

but not `scope_key`:

- `apps/coding_agent/lib/coding_agent/tools/search_memory.ex:38`

Execution resolves scope like this:

- session scope uses the injected `session_key`
- agent/workspace scopes try to read `scope_key` from params

See:

- `apps/coding_agent/lib/coding_agent/tools/search_memory.ex:79`
- `apps/coding_agent/lib/coding_agent/tools/search_memory.ex:101`

But because `scope_key` is not in the schema, models have no supported way to provide it. That leaves `scope_key = nil` for `agent` and `workspace`.

`MemoryStore.do_search/5` then falls through to the catch-all branch and executes `search_all`:

- `apps/lemon_core/lib/lemon_core/memory_store.ex:818`
- `apps/lemon_core/lib/lemon_core/memory_store.ex:832`

So a request like:

```json
{ "query": "deploy steps", "scope": "workspace" }
```

can search the entire memory DB instead of the workspace.

Why tests miss this:

- store tests only validate scoped search when an explicit `scope_key` is provided:
  - `apps/lemon_core/test/lemon_core/memory_store_test.exs:184`
- there is no tool-level test for `search_memory`.

### 3. High: the Python CLI launcher is broken and cannot start from the shipped script

The new launcher script under `clients/lemon-cli/bin/lemon-cli` resolves paths incorrectly.

It sets:

- `LEMON_ROOT` to `clients/lemon-cli`
- then `CLI_DIR="$LEMON_ROOT/clients/lemon-cli"`

See:

- `clients/lemon-cli/bin/lemon-cli:6`
- `clients/lemon-cli/bin/lemon-cli:9`

That produces a nonexistent nested path:

```text
.../clients/lemon-cli/clients/lemon-cli
```

The script then does:

- `clients/lemon-cli/bin/lemon-cli:26`

which should fail before the client starts.

Why tests miss this:

- there are no Python client tests
- current CI/quality checks do not exercise this launcher path.

### 4. High: the Python WebSocket client is not protocol-compatible with the existing control-plane flow

The WebSocket transport in `clients/lemon-cli` does not appear compatible with the existing OpenClaw control-plane contract used by the TypeScript client.

Problems:

1. It never sends the initial `connect` request after opening the socket.
   - Python client:
     - `clients/lemon-cli/src/lemon_cli/connection/websocket.py:58`
     - `clients/lemon-cli/src/lemon_cli/connection/websocket.py:64`
   - Established TypeScript flow sends `connect` immediately:
     - `clients/lemon-tui/src/agent-connection.ts:796`

2. It uses different request method names than the existing client.
   - Python sends:
     - `chat.prompt`
     - `sessions.close`
     - `session.set_active`
   - See:
     - `clients/lemon-cli/src/lemon_cli/connection/websocket.py:263`
   - TypeScript sends:
     - `chat.send`
     - `sessions.delete`
     - `sessions.active`
   - See:
     - `clients/lemon-tui/src/agent-connection.ts:957`

3. The Python class comments say “OpenClaw protocol”, but the actual request/response surface does not match the existing implementation the repo already relies on.

This makes WS mode likely to connect but fail to authenticate, start sessions, or deliver prompts correctly.

Why tests miss this:

- there are no protocol or integration tests for `clients/lemon-cli`
- no cross-check against the existing TS client contract.

### 5. High: registry installs can incorrectly elevate community skills to `:official` trust

The registry source documents namespace-based trust:

- `official/...` => `:official`
- everything else => `:community`

See:

- `apps/lemon_skills/lib/lemon_skills/sources/registry.ex:8`
- `apps/lemon_skills/lib/lemon_skills/sources/registry.ex:128`

Search results correctly construct entries with namespace-derived trust:

- `apps/lemon_skills/lib/lemon_skills/sources/registry.ex:131`

But install planning ignores namespace-specific trust and instead always uses:

- `mod.trust_level()`

See:

- `apps/lemon_skills/lib/lemon_skills/installer.ex:156`

The registry source module hardcodes:

- `apps/lemon_skills/lib/lemon_skills/sources/registry.ex:95`

to `:official`.

So `community/...` refs routed through `SourceRouter`:

- `apps/lemon_skills/lib/lemon_skills/source_router.ex:74`

can be installed with `:official` trust, bypassing the community trust posture described in the module docs and likely affecting audit expectations.

Why tests miss this:

- routing tests cover registry refs, but not the persisted `trust_level` after install
- installer tests do not assert audit/trust behavior for non-`official` registry namespaces.

### 6. Medium: `mix lemon.update` claims to migrate deprecated config, but `[agent]` is not actually migrated

`ConfigMigrator.check/1` correctly reports `[agent]` as deprecated:

- `apps/lemon_core/lib/lemon_core/update/config_migrator.ex:91`

But `migrate!/1` only performs text replacements for:

- `[agents.*] -> [profiles.*]`
- `[tools.*] -> [runtime.tools.*]`
- `[agent.tools.*] -> [runtime.tools.*]`

See:

- `apps/lemon_core/lib/lemon_core/update/config_migrator.ex:126`
- `apps/lemon_core/lib/lemon_core/update/config_migrator.ex:133`
- `apps/lemon_core/lib/lemon_core/update/config_migrator.ex:143`

There is no migration for `[agent]` itself.

That matters because modular config loading still hard-fails on `[agent]`:

- `apps/lemon_core/lib/lemon_core/config/modular.ex:97`
- `apps/lemon_core/lib/lemon_core/config/modular.ex:220`

So a user can run the migrator successfully, keep a still-deprecated `[agent]` section, and continue to get config load failures.

Why tests miss this:

- migrator tests only assert `[agents.*]` and `[tools.*]` rewrites:
  - `apps/lemon_core/test/lemon_core/update/config_migrator_test.exs:61`
- there is no success-path test for a config containing `[agent]`.

### 7. Medium: skill updates do not persist refreshed `upstream_hash`

The update path checks drift using `upstream_hash`, and if content changed it runs a reinstall:

- `apps/lemon_skills/lib/lemon_skills/installer.ex:326`

When the new upstream hash differs, it does:

- `apps/lemon_skills/lib/lemon_skills/installer.ex:332`

and only after reinstall returns does it update the returned struct with:

```elixir
%{updated | upstream_hash: upstream}
```

However, the reinstall has already written the updated entry to:

- lockfile
- registry

using the pre-hash struct:

- `apps/lemon_skills/lib/lemon_skills/installer.ex:372`
- `apps/lemon_skills/lib/lemon_skills/installer.ex:373`

So the caller receives the new hash, but persisted state may still carry the old one. That can break “already up to date” detection and cause redundant re-fetching.

Why tests miss this:

- there is no successful update-path test asserting persisted provenance after update
- current tests do not inspect lockfile/registry state for refreshed `upstream_hash`.

### 8. Medium: runtime boot does not actually enforce the `product_runtime` feature gate

The new runtime boot docs say boot should check whether `product_runtime` permits startup:

- `apps/lemon_core/lib/lemon_core/runtime/boot.ex:14`

The feature defaults to `off`:

- `apps/lemon_core/lib/lemon_core/config/features.ex:11`

But `Boot.start/2`:

- loads dotenv
- applies ports
- checks whether something is already running
- starts the requested profile

without checking the feature flag at all:

- `apps/lemon_core/lib/lemon_core/runtime/boot.ex:42`

That makes the documented rollout gate ineffective.

Why tests miss this:

- there are tests for `Env`, `Health`, and `Profile`
- there is no boot test covering `product_runtime` gating behavior.

## Additional Noted Risks

### Python CLI WebSocket config shape bug

`ControlPlaneConfig.scopes` is loaded as `list[str] | None`:

- `clients/lemon-cli/src/lemon_cli/config.py:69`
- `clients/lemon-cli/src/lemon_cli/config.py:299`

But `main()` assumes `config.ws_scopes` is a string and calls:

- `clients/lemon-cli/src/lemon_cli/cli.py:48`

That will raise `AttributeError` when scopes are configured as a TOML list, which is the natural representation for scopes in config.

### Python CLI WebSocket flags are parsed but mostly ignored

CLI flags are accepted for:

- `--ws-token`
- `--ws-role`
- `--ws-scopes`
- `--ws-client-id`

See:

- `clients/lemon-cli/src/lemon_cli/cli.py:21`

But `resolve_config()` only applies env/config values for those fields and does not read the CLI arguments for them:

- `clients/lemon-cli/src/lemon_cli/config.py:294`
- `clients/lemon-cli/src/lemon_cli/config.py:300`

That means CLI behavior contradicts the help text.

### Windows path-boundary checks are POSIX-only

Both `read_skill` and skill lint use:

```elixir
String.starts_with?(expanded, expanded_skill <> "/")
```

See:

- `apps/lemon_skills/lib/lemon_skills/tools/read_skill.ex:323`
- `apps/lemon_skills/lib/lemon_skills/audit/skill_lint.ex:164`

That logic is not portable to Windows paths and can reject valid files under the skill directory on `win32`.

### Release workflow manual dispatch likely uses the wrong tag

The release workflow resolves tag name as:

- `github.ref_name || github.event.inputs.tag`

See:

- `.github/workflows/release.yml:57`
- `.github/workflows/release.yml:233`

On `workflow_dispatch`, `github.ref_name` is typically the branch name, so it can override the user-supplied tag and point validation/release creation at the wrong ref.

## Assumptions / Open Questions

### Routing feedback

This review assumes the intended behavior is for `routing_feedback` to influence normal tool-using runs, not just zero-tool/chat-like runs. If the design intentionally wants pre-execution lookup with an empty toolset, then ingest should likely not persist toolset in the routing key.

### Registry trust

This review assumes registry namespace is part of the trust boundary. If all registry refs are intended to be trusted equally, then current documentation and install metadata should be updated to reflect that.

## Residual Risk

The biggest residual risk after these findings is the new Python client surface. It introduces a new runtime, transport, and launcher path with effectively no automated coverage in this PR, so additional defects in lifecycle, protocol mapping, or packaging are still plausible even beyond the issues listed here.
