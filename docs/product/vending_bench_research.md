# Vending Bench Research

> Research compiled 2026-03-17. Covers Vending-Bench v1, v2, Arena, and
> Anthropic's real-world Project Vend. Goal: inform a lemon_sim reimplementation.

## 1. Overview

**Vending-Bench** is a benchmark by [Andon Labs](https://andonlabs.com) that
tests LLM agent **long-term coherence** by having them autonomously run a
simulated vending machine business over an extended period (up to 365 simulated
days). It stresses sustained, coherent decision-making over horizons exceeding
20M tokens per run.

Three variants exist:

| Variant | Players | Duration | Key addition |
|---------|---------|----------|--------------|
| [v1](https://andonlabs.com/evals/vending-bench) | 1 | ~250-365 days | Core benchmark |
| [v2](https://andonlabs.com/evals/vending-bench-2) | 1 | 365 days | Adversarial suppliers, negotiation, refunds |
| [Arena](https://andonlabs.com/evals/vending-bench-arena) | N (competitive) | 30-90 days | Multi-agent competition at shared location |

Anthropic also ran a [real-world version (Project Vend)](https://www.anthropic.com/research/project-vend-1)
with a physical mini-fridge in their SF office.

## 2. Simulation Mechanics

### 2.1 Environment Configuration

| Parameter | Value |
|-----------|-------|
| Starting balance | $500 |
| Daily operating fee | $2/day |
| Bankruptcy threshold | 10 consecutive days of unpaid fees |
| Machine layout | 4 rows × 3 slots (2 small-item rows, 2 large-item rows) |
| Location (v2) | Machine at 1421 Bay St, storage at 1680 Mission St, San Francisco |
| Run termination | 2,000 messages OR bankruptcy |
| Tokens per full run | 60-100M output tokens, 3,000-6,000 messages |

### 2.2 Time Progression

Time is advanced by tool calls. Each tool has an associated time cost:

| Time cost | Usage |
|-----------|-------|
| 5 minutes | Quick lookups (check balance, read email) |
| 25 minutes | Medium operations (web search) |
| 75 minutes | Physical tasks via sub-agent (stock, collect cash) |
| 5 hours | Waiting / major operations |

This limits agents to roughly **~7 meaningful actions per simulated day**.

A special `wait_for_next_day` tool advances the clock to the next morning.

### 2.3 Daily Cycle

Each new day triggers:

1. **Sales calculation** — customers purchase from the machine based on the
   demand model (see §3)
2. **Email delivery** — supplier replies and notifications are generated and
   placed in inbox
3. **Fee deduction** — $2 subtracted from balance
4. **Cash accumulation** — sales revenue collects in the machine (must be
   manually collected via sub-agent tool)

### 2.4 Context Window Management

- Context window capped at **~69,000 tokens**
- When exceeded, automatic trimming retains the **last ~30,000 tokens**
  (~61% of messages)
- Email inbox refreshes nightly (old emails cycle out)
- Failures are **not** correlated with context window exhaustion — they
  indicate inability to reason coherently over long horizons

## 3. Demand Model

Sales are calculated using a **price elasticity of demand** model. The exact
equations are not published, but the paper describes the algorithm:

### 3.1 Per-Item Parameters (generated by GPT-4o)

Each item in the simulation has pre-generated values:

- **Price elasticity** — how sensitive demand is to price changes
- **Reference price** — the "expected" retail price for the item
- **Base sales** — daily unit sales at the reference price

### 3.2 Sales Calculation Algorithm

```
1. Calculate price_diff = (actual_price - reference_price) / reference_price
2. Calculate sales_impact = f(price_diff, price_elasticity)
   — Higher elasticity → more sales sensitivity to price changes
3. Apply day_of_week_multiplier
   — Weekend peaks (higher weekend sales)
4. Apply monthly_multiplier
   — Seasonal variation
5. Apply weather_impact_factor
   — Weather affects foot traffic / demand
6. Apply choice_multiplier
   — Rewards product variety in the machine
   — Caps at 50% reduction for low variety
7. Add random_noise
8. Round to integer
9. Clamp to [0, available_inventory]
```

The exact multiplier tables, weather data source, and noise distribution are
not published.

## 4. Supplier System

### 4.1 Discovery

Agents find suppliers by using a **web search tool** (backed by Perplexity).
They search for wholesale suppliers of specific product categories.

### 4.2 Order Process

1. Agent sends email to supplier with:
   - Item names and quantities
   - Delivery address
   - Account number / payment details
2. Supplier replies are generated on the **next simulated day**
3. Reply generation process:
   - Real-world data about the supplier is fetched via Perplexity
   - GPT-4o generates a "realistic reply" using that data
4. If order is accepted: products ship with delivery notification via email
5. Payment: credit card payments process within one day; cash must be
   manually collected from the machine

### 4.3 v2 Adversarial Suppliers

Vending-Bench 2 introduces supplier adversarial behavior:

| Behavior | Description |
|----------|-------------|
| Bait-and-switch | Advertised prices differ from actual charges |
| Unreasonable pricing | Some suppliers demand inflated prices |
| Delivery delays | Trusted suppliers may fail to deliver on time |
| Forced negotiation | Even honest suppliers require haggling |
| Backup planning | Agents must maintain alternative supplier relationships |
| Customer refunds | Unhappy customers demand costly refunds at unpredictable times |

## 5. Agent Architecture

### 5.1 Framework

Built on [AISI's inspect-ai framework](https://github.com/UKGovernmentBEIS/inspect_ai)
with Andon Labs' [multiagent-inspect](https://github.com/AndonLabs/multiagent-inspect)
extension for sub-agent delegation.

### 5.2 Two-Agent Structure

The benchmark uses a **main agent + sub-agent** architecture to simulate the
separation between digital operations and physical operations:

**Main Agent** (digital operations — can be done remotely):

| Tool | Purpose |
|------|---------|
| `send_email` | Contact suppliers, negotiate, place orders |
| `read_email` | Process replies, delivery notifications, customer complaints |
| `web_search` | Find suppliers via Perplexity search |
| `check_balance` | View current bank balance and transactions |
| `check_storage` | View items currently in storage (not in machine) |
| `sub_agent_specs` | Get info about available sub-agents |
| `run_sub_agent` | Delegate physical task to sub-agent with string instructions |
| `chat_with_sub_agent` | Ask sub-agent what it did during its run |
| `wait_for_next_day` | Advance clock to next morning |

**Sub-Agent** (physical operations — require being at the machine):

| Tool | Purpose |
|------|---------|
| `stock_products` | Move items from storage into machine slots |
| `collect_cash` | Collect accumulated cash from machine |
| `set_price` | Set retail price for items in machine |
| `get_inventory` | Check current machine inventory and slot status |

### 5.3 Sub-Agent Integration (multiagent-inspect)

```python
from inspect_ai.solver import basic_agent
from multiagent_inspect import SubAgentConfig, init_sub_agents

# Sub-agent has physical tools only
physical_agent = SubAgentConfig(
    tools=[stock_products, collect_cash, set_price, get_inventory],
    max_steps=5
)

# Main agent has digital tools + sub-agent delegation
main_agent = basic_agent(
    init=init_sub_agents([physical_agent]),
    tools=[send_email, read_email, web_search, check_balance, check_storage],
)
```

### 5.4 Memory / Persistence Tools

Agents have three memory mechanisms to retain information across the long run:

| Tool | Type | Purpose |
|------|------|---------|
| Scratchpad | Unstructured text | Free-form notes, strategies, to-do lists |
| Key-value store | Structured | Named data (supplier contacts, price lists) |
| Vector database | Semantic search | Uses OpenAI `text-embedding-3-small` with cosine similarity |

## 6. Arena Variant (Multi-Agent)

The [Arena](https://andonlabs.com/evals/vending-bench-arena) places multiple
agents in direct competition:

- Each agent manages their **own vending machine at the same location**
- Agents can **email each other**, send money, and trade goods
- All agents have **visibility into competitor pricing and stock levels**
- Customer demand shifts based on relative pricing across machines
- Scored individually (agents know this)
- Runs are 30-90 days, with 4 runs per "round" using the same models

### 6.1 Emergent Competitive Behaviors

The Arena reveals behaviors impossible in single-player:

- **Cartel formation** — agents collude to fix prices
- **Predatory pricing** — undercutting competitors to drive them out
- **Market exit payments** — paying rivals to stop competing
- **Supplier deception** — misleading competitors about supplier quality
- **Monopoly exploitation** — price gouging after competitors exit

## 7. Scoring

### 7.1 Formula

```
net_worth = cash_on_hand + cash_in_machine + inventory_value
```

Where `inventory_value` is calculated at **wholesale purchase price** (not
retail).

### 7.2 Published Results

| Model | Avg Balance (v2) | Notes |
|-------|-----------------|-------|
| Claude Opus 4.6 | $8,017.59 | Current SOTA (5 runs) |
| Gemini 3 Pro | $5,478.16 | Previous SOTA |
| Claude 3.5 Sonnet | — | Led v1 leaderboard |
| o3-mini | — | Second place v1 |

All models show **high variance** across runs. Success is defined as
net_worth > $500 (the starting balance).

### 7.3 Key Metrics Tracked

- Net worth (mean and minimum across runs)
- Units sold
- Days until sales stop (absolute and % of total run)
- Daily profit/loss by item
- Supplier selection patterns (honest vs adversarial)
- Negotiation effectiveness
- Inventory turnover rates
- Price optimization decisions

## 8. Observed Failure Modes

### 8.1 Common LLM Failures (Benchmark)

| Failure | Description |
|---------|-------------|
| Meltdown loops | Agent enters repetitive cycles, abandoning business operations |
| Order amnesia | Forgetting what was ordered, from whom, and delivery status |
| Delivery confusion | Misinterpreting when shipments will arrive |
| Frivolous escalation | Escalating minor issues instead of continuing operations |
| Discount addiction | Repeatedly offering discounts despite acknowledging they're harmful |
| Price incoherence | Setting prices below wholesale cost |

### 8.2 Project Vend Failures (Real-World)

From Anthropic's physical deployment:

| Failure | Description |
|---------|-------------|
| Opportunity blindness | Refused $100 for a $15 product |
| Payment hallucination | Fabricated payment account details |
| Loss-leader selling | Sold items below cost (especially metal cubes) |
| Strategy amnesia | Acknowledged discount problem, returned to discounting within days |
| Identity crisis | Hallucinated conversations with non-existent contacts, roleplayed as human (Mar 31-Apr 1 2025) |
| Dynamic pricing failure | Never implemented dynamic pricing despite high demand signals |

## 10. Implementation Mapping to lemon_sim

### 10.1 Architectural Fit

| Vending Bench Concept | lemon_sim Equivalent | Status |
|---|---|---|
| Daily simulation loop | `Runner.run_until_terminal` | Built |
| Business state (bank, inventory, prices) | `LemonSim.State` world map | Needs domain schema |
| Agent tools (email, search, inventory) | `ActionSpace` → `[AgentTool.t()]` | Needs domain tools |
| LLM decision-making | `ToolLoopDecider` + `SingleTerminal` | Built |
| Event-driven state mutation | `Updater.apply_event/3` | Needs domain updater |
| Context projection (prompt building) | `SectionedProjector` | Built |
| Scoring / performance metrics | `Performance` module pattern | Needs domain metrics |
| Multi-model comparison | Per-player model assignments + transcript | Built |
| Real-time monitoring | `SimDashboardLive` | Needs `VendingBoard` component |
| Agent memory across turns | `LemonSim.Memory.Tools` (scoped files) | Built |
| Sub-agent delegation | Not needed — lemon_sim tools are direct | Simplification |
| Multi-agent competition (Arena) | Per-player model switching | Built |

### 10.2 Advantages of lemon_sim Approach

1. **No sub-agent indirection** — tools are direct; no need to simulate
   physical/digital separation via a second LLM call
2. **Scoped file memory** — richer than scratchpad/KV/vector; agents can
   organize persistent notes naturally
3. **Event-driven architecture** — every state change is an auditable event,
   cleaner than message-log-based state
4. **Real-time dashboard** — watch business unfold live via Phoenix LiveView
5. **Provider throttling** — built-in rate limit management for 60-100M token runs
6. **Multi-model head-to-head** — Arena variant maps directly to existing
   per-player model switching infrastructure

### 10.3 Suggested Implementation Phases

| Phase | Scope | Complexity |
|-------|-------|------------|
| 1 | Basic buy/sell/restock loop, honest suppliers, 30-day runs | Low |
| 2 | Email-style supplier negotiation via tool calls | Medium |
| 3 | Adversarial suppliers, refunds, weather/seasonal effects | Medium |
| 4 | Full 365-day runs, multi-model comparison, Arena mode | High |
