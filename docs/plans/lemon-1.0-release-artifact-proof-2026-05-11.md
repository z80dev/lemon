# Lemon 1.0 Release Artifact Proof - 2026-05-11

This records a local release-artifact install proof for the Lemon 1.0 launch
readiness goal. It uses the same artifact shape as `.github/workflows/release.yml`:
a tarball of the assembled release directory plus a manifest entry with size and
SHA-256.

This is not a public GitHub Release proof and not a clean OS/container proof. It
proves that the current Linux `lemon_runtime_min` and `lemon_runtime_full`
artifact shapes can be packaged, verified, extracted into a clean prefix, booted
without Mix, and used to generate a release-runtime support bundle.

This local artifact proof was refreshed on 2026-05-12 after upgrading the
maintainer host to the supported 1.0 toolchain: Elixir 1.19.5 on Erlang/OTP
28.5 (ERTS 16.4). The supported GitHub release workflow is pinned to the same
Elixir/OTP pair, but this remains a local proof rather than public release
evidence.

## Commands

Build the release:

```bash
MIX_ENV=prod mix deps.get --only prod
MIX_ENV=prod mix clean
MIX_ENV=prod mix release lemon_runtime_min --overwrite
MIX_ENV=prod mix release lemon_runtime_full --overwrite
```

Package using the release workflow shape:

```bash
tar -czf lemon-2026.05.0-local-proof-linux-x86_64-<profile>.tar.gz \
  -C ./_build/prod/rel/<profile> .
```

Verify the checksum, extract the artifact, start the release daemon, poll
control-plane health, and generate a support bundle through release `eval`:

```bash
scripts/verify_release_runtime_boot /tmp/lemon-release-artifact-proof-2026-05-0/artifacts
```

The full profile requires `LEMON_WEB_SECRET_KEY_BASE` for release boot and
release `eval`, even for diagnostic commands, because `runtime.exs` configures
the Phoenix endpoint before the eval expression runs.
